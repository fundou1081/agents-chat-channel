# 2026-06-06 — 3-Channel Architecture (方案 B)

## 今日完成

把项目从 4 套机制 (DM + Bulletin + FreeChat + 新增 Channel) 合并成 **3 套机制** (DM + Posts + Channels)。

### 设计决策 (跟用户讨论的)

**用户提的问题**: 4 套独立 (A) vs 合并 Bulletin+FC (B) vs 全部 Channel 化 (C)

**从可维护性 + 信息流 2 个角度评估后, 选 B**:
- 保留 3 种本质信息流: DM (推-1) / Posts (拉-N) / Channels (推-N)
- Bulletin+FreeChat 合并成 Posts (一张表, 4 个 kind 共享字段)
- Channels 新增 (持久主题, 订阅推送)
- 减少 schema/API/测试 ~25%

### 关键设计原则 (沿用)

1. **Email Model (Pull + Burst)** — 持久可重放, 解耦 sender/receiver
2. **Author = 长生命周期 actor** — 长期活着, 多 session
3. **Per-Author Backend** — PM=qwen (快), 工程师=opencode (能改文件)
4. **Recipient Routing** — alias 表 + 模糊匹配, 防 LLM 输出错 id
5. **Network Policy** — 5 维限流, 防烧钱/暴走
6. **Monitor** — 完整事件流, 不可变 jsonl
